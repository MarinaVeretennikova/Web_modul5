Установка Apache 2.0

Автор: Ryan Bloom.
Перевод: ApacheDev.ru.

Оригинал статьи

Редакторский комментарий: Райн Блум (Ryan Bloom) был одним из основных разработчиков сервера Apache 2.0. Мы встретили его на выставке Linux World в Нью-Йорке и попросили его написать серию статей, описывающих новую версию Apache, для наших читателей. Это первая его статья, рассказывающая, что нового появилось в Apache 2.0 и как его установить.

Разработчики Apache работали над Apache 2.0 последние 2 года, и 4 Апреля первая бета версия Apache 2.0 была выпущена. Это первая из серии статья, в которой мы объясняем чем Apache 2.0 отличается от Apache 1.3 и чего пользователям сервера следует опасаться.

Зачем делать апгрэйд?

Одно из самых важных изменений - это добавление поддержки многонитеевость в Apache 2.0. Версия 1.3 использовала потоки для обработки запросов, и это существенно повышало производительность на многих операционных системах. Нити же, совместно с потоками, делают Apache 2.0 более масштабируемой.

Второе значимое отличие от Apache 1.3 - это фильтрация ввода/вывода, которая позволяет одному модулю изменять вывод другого модуля. Такая возможность была необходима в Apache 1.3 для CGI скриптов с SSI тэгами. Но это не было возможно в версии 1.3, но благодаря фильтрации ввода/вывода это стало возможным при работе с Apache 2.0.

Apache 2.0 также предоставляет поддержку SSL без модификации ядра сервера. Минимально необходимая поддержка SSL реализована в модуле mod_tls, которая в недалеком будущем расшириться до полной поддержки SSL.

Первая трудность при использовании Apache 2.0, это инсталляция и конфигурация сервера Apache. Есть две формы конфигурации сервера: при компиляции и перед запуском. Эта статья описывает получение дистрибутива, его компиляцию и установку сервера.

Установка и конфигурация сервера Apache 2.0

Перед тем как мы начнем, скачайте последнюю версию сервера Apache 2.0. Наилучшее место для получения стабильной копии Apache - это официальный сайт веб сервера. Если вы разработчик, и вам потребовалась последняя разработанная версия, то вам следует отправиться на страницу разработчиков Apache, и почитать, перейдя по ссылкам, о получении кода из CVS.

Это статья предполагает, что вы используете дистрибутив Apache 2.0.15, хотя в будущих версиях Apache все должно выглядеть похожим.
Сейчас вы получили дистрибутив, то распакуйте его, используя команду:

tar -xvzf httpd-2*.tar.gz

После этого мы можем приступить к конфигурации Apache. В предыдущих версиях Apache, были две различные модели настройки сервера, каждая из которых развивалась отдельно. Первый метод был использование текстового файла, контролирующего состав сборки. Второй была autoconf - подобная система, которая реально только модифицировала тестовый файл под себя.

В Apache 2.0, разработчики решили, что более интересно написание веб-сервера, который поддерживает второй метод настройки. Поэтому, Apache 2.0 использует autoconf и libtool для определения компонентов сборки. Дистрибутивы, скачанные с сайта Apache, уже имеют созданные конфигурационные скрипты, но для тех, кто скачал разрабатываемую версию, необходимо сгенерировать конфигурационный скрипт командой ./buildconf. Данная команда проверить все директории дистрибутива, чтобы убедиться в том, что у вас есть все необходимые компоненты для Apache.

После этого, у вас должен появиться конфигурационный скрипт в корневом каталоге дистрибутива Apache. Это скрипт, который фактически сконфигурирует сервер Apache. Тем не менее, данный скрипт имеет множество опций, которые позволят пользователю контролировать каждый аспект сборки Apache. Полный список этих опций можно получит выполнив команду: ./configure --help.

Обсуждение каждой опции выходит за рамки данной статьи, но я опишу несколько наиболее простых и важных опций.

  • --prefix. Определяет место инсталляции сервера apache.
  • --with-maintainer-mode. Компиляция с отлаживаемой информацией. Данная опция будет оставаться важной, по крайней мере, до тех пор, пока не будет стабильной версии сервера.
  • --with-mpm. Определяет модуль поддержки многопроцессности, который будет использоваться. Многопроцессность - это встроенная возможность Apache. Более полное описание этих возможностей сервера могут быть найдены в онлайне.
  • --enable-module. Определяет какие модули будут встроены в ядро Apache. Это может быть либо список модулей, либо слово most, которое задает минимально необходимый список модулей.
  • --enable-mods-shared. Определяет какие модули следует компилировать отдельно от сервер

    Многопроцессные модули (MPMs) сервера позволяют вам настроить Apache 2.0 для любого сайта. Что работает на одном сайте или операционной системе, может некорректно работать при другом окружении. Чтобы решить эту проблему, системные администраторы могут выбрать режим работы сервера. Некоторые операционные системы поддерживают больше режимов работы сервера, чем другие. Например, У Unix есть 3 стандартных MPM:

    Prefork - модель, как у Apache 1.3. Родительский процесс создает множество дочерних процессов, которые обрабатывают запросы. Каждый дочерний процесс имеет одну нить и может обработать только один запрос за раз. Когда нагрузка увеличивается, сервер создает новые дочерние процессы для обработки поступающих запросов.
    Threaded - модель, подобная Prefork, за исключением того, что каждый дочерний процесс имеет постоянное количество нитей. Точное количество нитей определяется при запуске из файла httpd.conf
    Perchild - Родительский процесс создает определенное количество дочерних процессов, каждый с минимальным количеством нитей. Когда нагрузка на сервер возрастает, потоки создают новые нити, чтобы обработать возрастающий трафик.

    С другой стороны, Windows имеет только одну опцию.

    Winnt - модель, как у Apache 1.3, где только два процесса: дочерний, обрабатывающий запросы и родительский, отвечающий за контроль над дочерним процессом для уверенности в постоянной обработке запросов.

    Компиляция сервера.

    После того, как выполнился конфигурационный скрипт, необходимо скомпилировать сервер. Это делается с помощью команды make в корневой директории исходников сервера. Последний шаг - это фактически установить новый сервер, что делается командой make install. Эта команда скопирует все необходимые файлы в директорию, определенную опцией -prefix конфигурационного скрипта. Если данная опция не была указана, то путь /usr/local/apache/ будет использоваться по умолчанию.

    Вот мы почти все и сделали. Если установка прошла успешно, то необходимо перейти в созданную директорию и запустить сервер командой ./bin/apachectl start. Тем не менее, нам необходимо знать какой порт прослушивает сервер. Если вы устанавливали сервер под пользователем root, то сервер слушает 80 порт, как стандартный HTTP порт, но если вы производили установку под другим пользователем, то сервер работает с портом 8080. Номер порта может быть изменен в файле httpd.conf в директиве Port. Итак, запустите свой любимый браузер и запросите страницу с адресом, либо http://localhost/, или http://localhost:8080/, в зависимости от используемого порта. Вы должны увидеть стандартную веб-страницу Apache. Если этого не произошло, то журнал ошибок расскажет вам в чем проблема.

    Вы только что настроили ваш первый сервер Apache 2.0 и готовы обрабатывать веб-страницы. Однако помните, что данный сервер пока не является релизом, поэтому иногда могут произойти ошибки. Если вы их обнаружили, пожалуйста, отправьте их описание в базу ошибок сервера.
